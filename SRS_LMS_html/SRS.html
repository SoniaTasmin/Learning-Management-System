1. Introduction


1.1 Purpose

This document will provide a comprehensive description of Learning Management System. The purpose of this document is to provide work analysis of the system features and interface.

1.2 Document Conventions

We’re following the IEEE standards in writing this SRS document. Every requirement statement is to have its own priority.

1.3 Intended Audience and Reading Suggestions

This document is restricted to developers and coworkers.

1.4 Product Scope

Learning Management System is an online teaching platform. Its purpose is to provide a common platform between teachers and students. All the possible teaching material will be provided in the platform. Teachers can create classroom and the system will generate a classroom code. Students can join classroom using classroom code. Teachers can also assign class materials. Students can preview the class materials. Students can ask questions in different ways(written and verbal). System can identify students via video authentication.


1.5 References

N/A


2. Overall Description


2.1 Product Perspective.

The system is a new product and self-contained product. It consists of a website and a mobile app integrated to a database. The database stores following information.

Teacher details:

It includes the name, email of a teacher, along with the listed students to a specific teacher.

Student details:

It includes username, first name, last name of a student along with their enroll courses.

2.2 Product Functions
The Cryo has the following functionalities:
  • Teacher Registration
  • Teacher Login
  • Student Registration
  • Student Login
  • Teachers can create classroom
  • Teachers can add students to classroom
  • Student can record questions
  • System can take photograph of students and verify them
2.3 User Classes and Characteristics
Teacher Can create class, assign students to a classroom and provide teaching materials.
Student Can preview classroom materials and zoom-in and zoom-out materials.
System Controls authentication, data storage, indexing

2.4 Operating Environment
  • Operating System: Windows/Android
  • Platform: Website and Android application
  • Database: MySQL
  • Client/server system
  • Languages: Java, Python
  • Framework: Django
2.5 Design and Implementation Constraints
  • Using a framework
  • Using Java/Kotlin for the app
  • Designing by firebase

N/A

2.7 Assumptions and Dependencies
  • Full functional rest API
  • Implementing libraries
  • Availability of resources

4. System Features


4.1 User Registration and Login
4.2.1 Priority: HIGH

Description: Upon visiting the website, the user can browse through the list of items

4.1.2 Stimulus/Response Sequences
  • Enter the app or website
  • Register into app or website.
  • If already a registered user, then login
  • Autheticate users and then redirect to homepage
4.1.3 Fucntional Requirements
  • Incorrect login information will prompt the system to the user to enter correct credentials.
4.2 Classroom create and join
4.2.1 Priority: HIGH
  • Description: Teacher can create classroom. Teacher can share classroom code with students and students can join classroom. Classroom details will be saved on database.
4.2.2 Stimulus/Response Sequences
  • Teacher will click on the create classroom button
  • System will generate unique classroom code
  • Using the classroom code students can join the classroom.
4.1.3 Fucntional Requirements
  • If classroom code is invalid system prompts students to enter correct classroom code.
4.3 Assign classroom materials
4.3.1 Priority: LOW

Description: Teachers can assign reading materials in classroom. Registered users can view reading materials by clicking on it.

4.3.2 Stimulus/Response Sequences
  • Login to the system
  • Teachers assigning materials
  • Registered users can preview the materials
4.1.3 Fucntional Requirements
  • If login credentials are invalid system prompts user to enter correct credentials.
  • If no materials are provided the section will be empty.
4.4 Take Photograph and authentication
4.4.1 Priority: HIGH
  • Description: System will take photograph of student within a time interval. The photograph will be saved on database and the saved photograph will be used to authenticate students.
4.4.2 Stimulus/Response Sequences
  • Login to the app/website
  • Click on the reading materials
  • System takes photograph of the student
  • Authenticate the student
4.1.3 Fucntional Requirements
  • If login credentials are invalid system prompts user to enter correct credentials.
  • If the server fails to function the system would wait for the server to recover.
4.5 Asking Questions
4.5.1 Priority: MEDIUM
  • Description: Student can ask questions in various ways. Questions can be asked verbally, written down or tagged in a particular part of the material.
4.5.2 Stimulus/Response Sequences
  • Login to the system
  • Click on the reading material
  • Ask questions in any preferrable way
4.1.3 Fucntional Requirements
  • If login credentials are invalid system prompts user to enter correct credentials.
  • If the server fails to function the system would wait for the server to recover.

6. Other Nonfunctional Requirements


6.1 Performance Requirements

It provides a detailed analysis of the user interaction with the software and measurements placed on the system performance. The performance requirements are:

  • The software will load and be usable within 3 seconds to make it more flexible
  • The software will update the interface depending on any interactions within 2 seconds for a good performance speed
  • The database will be normalized to prevent redundant data and improve performance
6.2 Safety Requirements

To ensure that there is no loss of data due to any external or internal issues, the data will be stored in cloud storages. This allows a safe storage as well as easy accessibility

6.3 Security Requirements

The system will not disclose any personal information about the users/clients. The software will not grant any access to an unauthorized user and will not communicate with any other software or devices during its usage. The image reels that will be taken when the student views a document will only be accessible by the teacher associated with that student. No other user will have access to those images.

6.4 Software Quality Attributes
  • AVAILABILITY: The website is accessible as both a desktop version and as an android version. It is very user-friendly and easy to handle.
  • RELIABILITY: The website will ensure no data is leaked. All images and personal information will be handled with care.
  • FLEXIBILITY: As it is user-friendly, the student will be able to manage their classes and the progression of their reading easily. No complex operations will be required to manage their reading materials. The teacher can also access the records of the students in whatever form he wants. The sorting options will ensure that the teacher can get to where he wants easily and without any complications.
  • USABILITY: Users with basic English knowledge will be able to access this website and use it easily.
6.5 Business Rules

This is a free software and will not endorse any adverts.
